## 一、这个项目是什么(一页讲清)

**句子秒聘 = 一个上岗在真实生产环境的 AI 招聘专员。**

它接管了招聘流程里所有"搬信息"的活,HR 只做三次拍板(约不约面、触不触达、时间行不行),其余全自动:

```
BOSS直聘来的简历 → 自动进飞书表格建档 → HR点约面 → AI加候选人企微
→ AI对话(答疑/约时间/改期协调) → 自动建日程开录制 → 面试后AI按公司标准写面评
```

**为什么它能拿冠军**:别的队伍做的是演示品(demo),它是真的在公司生产环境跑着、处理过真实候选人的系统。"别的项目做完,比赛就结束了;我们的项目做完,它明天照常上班。"

**核心设计哲学(商业化时就讲这三条)**:
1. **单一真源**:所有业务规则(岗位怎么打分、候选人问题怎么答、红线是什么)都不写在代码里,而是实时读 HR 知识库(飞书表格/文档)。**HR 改一行表格 = 给 AI 编程**,2分钟生效
2. **确定性编排 + LLM 分工**:流程用确定性的代码状态机(绝对稳),大模型只在三处出手:判岗位、识意图、写面评(足够聪明)
3. **长在已有工作场景里**:不让人迁移到新系统,AI 长在 HR 本来就在用的飞书表格上——这是和市面招聘系统(一年几十万、要全公司换工具)的本质区别

## 二、这件事是怎么一步步做出来的(时间线)

**第1步·从脚本到服务(7/17-7/18)**
先用脚本把"AI-HR表点约面→自动写进度表"跑通(判岗位、读简历、设面试官),验证可行后,立刻打包成 Docker 服务部署上公司服务器。**关键决策:绝不依赖任何人的个人电脑/个人登录**——用"句子秒聘"应用身份跑,谁离职都不影响。

**第2步·权限攻坚(7/18)**
真进度表开了"高级权限",光加应用为协作者不够,要在**高级权限设置里给应用的角色配"可编辑"**。这类权限问题反复出现(附件下载、建文档、通讯录),是飞书生态开发的最大暗礁。

**第3步·场景迭代(7/18-7/19,最重要的阶段)**
玄玄按真实使用场景逐个口述规格,AI 逐个实现:
- 手动丢简历→自动识别建档(含照片简历)
- 面评自动生成(嵌原始简历+按岗位标准包评分)
- 人工改岗位→面评自动联动
- 触达前自检(缺信息群里@人,不裸触达)
- 联系方式=微信号也能触达;改号自动重触达
- 改期协调(候选人给区间→@面试官拍板→改表格=确认→自动通知候选人)
- 转人工双向开关(勾复选框=人工接管,取消=AI恢复)
- 知识库飞书化(FAQ 176条进表格,AI 答疑实时读)
- 面后面评(贴逐字稿→按胜任力模型生成标准格式面评)

**第4步·真人实测暴雷再修(7/19)**
用玄玄本人当候选人全链路真测,暴露并修复了一批真问题(见第六章"踩过的坑")。**教训:demo 跑通≠能用,真人测试才见真章。**

**第5步·Demo 展演(7/19-7/20)**

## 三、系统全景(三个服务+一条穿透链)

### 三个服务(都跑在公司服务器的 Docker 容器里)

| 服务 | 容器名 | 谁写的 | 干什么 |
|---|---|---|---|
| **表格管理服务** | aihr-table | 玄玄+AI(Python) | 一切和飞书表格相关的自动化:约面同步/简历建档/面评生成/9条规则轮询 |
| **触达服务** | miaopin | 宏佳(NestJS),现共同维护 | 一切对外发消息+MongoDB:加企微好友/和候选人对话/建日程 |
| **运营中台** | miaopin-console | 宏佳 | 运营后台:看触达任务/漏斗统计/配置管理/HR名录维护 |

外加一个 **nginx**(容器 aihr-nginx):门卫,占80端口,按路径把请求分发给触达服务或中台。

**分工铁律**:动表格=表格服务;动 Mongo+对外发消息=触达服务。两个服务通过 HTTP 接口互相调用,谁也不越界。

### 五层穿透(demo 讲的核心技术,也是真实架构)

```
① BOSS直聘(简历来源) → ② 飞书多维表格(HR工位) → ③ 飞书(消息/日历/文档)
→ ④ 企业微信(触达通道) → ⑤ 候选人个人微信
Agent(大模型)贯穿:判断/对话/评价
```
五个系统分属三家公司生态,互为数据孤岛,被串成一条线——HR 只面对一张表格。

## 四、技术底层扫盲(非技术人员版,重点章节)

### 4.1 服务器是什么
就是一台**永远不关机的电脑**,放在云机房里,没有屏幕键盘,只能远程操作。我们租的这台在火山引擎(IP 地址记在服务器台账/HANDOFF,不进这份会公开的文档)。
- **登录方式**:SSH(一种远程命令行),凭一把"SSH 密钥文件"(在玄玄电脑的 ~/.ssh/ 里)登录,不用密码
- **上面跑着什么**:6个 Docker 容器(上面三个服务+nginx+两个数据库)
- **为什么要服务器**:服务要 7×24 跑,不能依赖谁的笔记本开着

### 4.2 Docker 容器是什么
把"程序+它需要的一切环境"打包成一个**标准集装箱**。好处:在谁的电脑上都一样跑,装卸(部署/回滚)都是分钟级。
- 常用操作(都是在服务器上执行):`docker ps` 看哪些容器在跑;`docker logs 容器名` 看某个服务的日志;`docker restart 容器名` 重启
- 我们的部署流程:改代码→提交到 GitHub→在服务器上重新"打包集装箱"(docker build)→换新箱子上岗(docker run)

### 4.3 端口是什么
一台服务器像一栋楼,**端口就是门牌号**,不同服务开不同的门:
| 端口 | 谁在用 | 说明 |
|---|---|---|
| 80 | nginx | 对外大门(ai-hr.juzibot.com 进来先到它),再按路径分发 |
| 8090 | 表格管理服务 | 只对内(触达服务调它的接口用) |
| 8091 | 运营中台后台 | 只对内(nginx 转发) |
| 27017 / 6379 | MongoDB / Redis | 数据库,只对内 |

**教训**:两个程序不能抢同一个门牌号——我们撞过一次80端口(nginx 上线时触达服务还占着80),所以后来定了"部署单人收口"。

### 4.4 密钥都有哪些(存放位置,不写明文!)
密钥=各系统的"身份钥匙",**绝对不能写进代码、不能进 GitHub、不能发群里**。这份文档也只写"钥匙放在哪",不写钥匙本身:

| 密钥 | 是什么 | 放在哪 |
|---|---|---|
| 飞书应用凭证(app_id + app_secret) | "句子秒聘"应用的身份证,一切飞书操作靠它 | 服务器两个服务目录的 .env 文件;飞书开放平台后台可查 |
| 火山豆包 ARK key | 调大模型的钥匙 | 服务器 .env;火山方舟控制台可查 |
| SSH 私钥 | 登服务器的钥匙 | 玄玄电脑 ~/.ssh/ |
| MongoDB/Redis 密码 | 数据库钥匙 | 服务器 .env |
| 秒回平台 token | 企微触达通道的钥匙 | 服务器 .env;中台配置页 |

**.env 文件**:每个服务目录下的一个小文件,专门放密钥和配置(比如轮询几秒、开不开演练模式)。**改 .env 要重启容器才生效**。

### 4.5 API 是什么 + 我们用的关键 API
API = 系统对外开放的"服务窗口",程序拿着密钥去窗口办事。我们用到的:

**飞书开放平台(用得最多)**
- 多维表格(bitable):读写记录、搜索、字段管理——整个表格自动化的基础
- 文档(docx):建面评文档、读知识库/逐字稿、往文档里插内容
- 消息(im):机器人往群里发消息/@人
- 日历(calendar):建面试日程+视频会议+自动录制
- 云文档权限(drive):附件上传下载、文档权限设置
- ⚠️ 常见坑:权限分"应用身份/用户身份"两栏,服务要开**应用身份**;open_id 每个应用不一样(A应用拿的id,B应用用不了);高级权限表要单独给应用角色授权

**火山方舟(豆包大模型)**
- 一个"聊天补全"接口,兼容 OpenAI 格式;同一个模型既能读文字也能看图(读照片简历就靠它)

**秒回(公司产品,企微触达)**
- 加好友接口、发消息接口、回调(好友通过了/候选人回消息了会"敲门"通知我们)

**服务之间的自有接口**(表格服务↔触达服务)
- /reach 发起触达、/handover 转人工开关、/notify 改期直推、/progress/backfill 回填进度、/progress/handover AI判定转人工、/kb 知识库

### 4.6 数据库是什么
- **MongoDB**:触达服务的记事本,存每个候选人的"触达任务"(加到好友没、聊到哪步、转人工没)
- **Redis**:临时便签,做"同一件事别重复干"的锁
- 表格服务没用数据库——**飞书表格本身就是它的数据库**(这是设计亮点:HR 看表格=看系统状态)

### 4.7 GitHub 协作(完整版见 7/20 对话,这里存精华)
- 仓库=项目云端总部;pull=下载最新;commit=打存档带说明;push=上传
- 代码不能像飞书文档实时协同,因为**改一半的代码跑不起来**,所以"改完打包再同步"
- PR(Pull Request)="我改好一版,请你过目再合入"——不熟的人之间用;自己人(协作者)直接推
- **铁律:仓库是唯一真相**——所有改动必须进仓库,部署只从仓库出(宏佳曾经改了线上没推仓库,导致判断全错)
- **五条协作规矩**:开工先pull/小步提交说明写人话/别同时改同一文件/大改走PR/部署单人收口(宏佳)

## 六、踩过的坑(每个都是真实学费)

1. **存量数据铁律**:新自动化规则上线,必须先把表里已有数据登记为"已处理"再启用——规则⑨没做这步,148条历史逐字稿被当新增,群里@同事刷屏,紧急停服。**"之前的数据都不要用新逻辑,新增的才在讨论范围"**
2. **真实测试才见真章**:LLM 的密钥一直没配,意图分类静默失败了两天没人发现(全靠模板消息撑着)——所有"看起来在工作"的东西都要抽查证据
3. **飞书权限的暗礁**:高级权限表要单独授权;应用建的文档 HR 默认打不开(要主动开"组织内可编辑");open_id 分应用;应用身份建的日程不能指定主持人(录制归属问题至今未解)
4. **别绕过仓库改线上**:仓库和线上脱节后,一切判断都会错
5. **两人动同一台服务器要错开**:撞过80端口、差点互相覆盖配置(Mongo配置优先于env,中台初始化写错会静默改变线上行为)
6. **删除类操作永远先确认**:真表删记录,列清单给玄玄点头才动手

## 七、商业化视角(下一阶段的弹药)

**卖点(展演验证过的)**:
- 已在真实生产环境上岗,不是概念
- 长在客户已有的飞书工位上,零迁移零学习(对比市面招聘系统:贵+要全员换工具)
- 单一真源:客户 HR 自己改表格就能调教 AI,不用求供应商
- 五层穿透链路(BOSS→表格→飞书→企微→个人微信)市面没有

**复制方法论(给下一个客户/下一个岗位)**:
换一套知识库(岗位标准包+FAQ表)+ 换一张表 = 另一个岗位的 AI 员工。架构、状态机、接口全部复用。

**现场反馈**:展演时有老板明确说"这是刚需,做好了我帮你们拉客户"。

**已知差距(卖之前要补的)**:多租户隔离、客户侧部署方案、妙记归属、微信号加友实测、二面三面轮次(已开发未启用)、数据看板。
